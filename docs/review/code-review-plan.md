# コードレビュー計画

最終更新: 2026-04-11

## 目的

- 現在の差分を、機能追加の正しさだけでなく回帰と運用リスクの観点でも確認する。
- レビュー順序を固定し、重要な変更から先に潰せるようにする。
- 指摘、保留事項、再確認項目を後追いできる形で残す。

## 対象範囲

今回のレビュー対象は、未コミット差分と新規追加ファイルを中心とする。

- `backend/`: API、認識ロジック、WebSocket、依存関係、設定
- `calc-service/`: ダメージ計算と統計ロジック、API、テスト
- `frontend/`: バトル画面、ダメージ表示、開発ツール、各種 Zustand ストア
- `data/`, `templates/`, `scripts/`: 生成物、参照データ、取得スクリプト
- `docs/`: 実装メモや調査資料で、レビュー判断に影響するもの

対象外:

- 既存の無関係な変更の修正
- リファクタだけを目的にした設計改善
- レビュー完了前の最適化作業

## レビュー方針

優先度は「壊れたときの影響が大きい順」にする。

1. サーバ境界と状態遷移
2. 計算結果の正しさ
3. UI とストアの整合性
4. 開発ツールと設定差分
5. データと生成物の妥当性

各レビューでは、見た目の好みより次を優先する。

- バグ
- 仕様逸脱
- 回帰
- 競合状態
- データ不整合
- テスト不足
- 運用負荷の増加

## レビュー単位

### 1. API / サーバ初期化

対象:

- `backend/app/main.py`
- `backend/app/dependencies.py`
- `backend/app/api/`

観点:

- ルーティング追加が既存 API と衝突していないか
- 依存性注入とアプリ起動時の初期化順が安全か
- 例外時のレスポンス、タイムアウト、バリデーションが十分か

完了条件:

- エンドポイント単位で入出力と失敗系を確認
- 追加 API に対する最小限のテスト有無を確認

### 2. 認識状態 / WebSocket / 対戦進行

対象:

- `backend/app/recognition/`
- `backend/app/ws/battle.py`
- `backend/config/regions.json`
- `backend/tests/test_scene_state.py`

観点:

- 状態遷移の破綻がないか
- フレーム処理や pause 制御で race condition が起きないか
- OCR 結果と対戦状態の対応関係が崩れないか
- 設定 JSON 変更が既存前提を壊していないか

完了条件:

- 状態遷移の主要パスを追跡
- 新規ロジックに対するテストと手動確認ポイントを記録

### 3. ダメージ計算機能

対象:

- `backend/app/api/damage.py`
- `backend/app/damage/`
- `calc-service/src/`
- `calc-service/tests/`

観点:

- バックエンドと計算サービスの契約が揃っているか
- パラメータ名、型、単位、デフォルト値の齟齬がないか
- 計算根拠のデータソースが一貫しているか
- 異常系と未対応ケースの扱いが明示されているか

完了条件:

- API 契約を往復で確認
- 少なくとも代表ケースのテスト有無を確認

### 4. フロントエンドの状態管理と UI

対象:

- `frontend/src/App.tsx`
- `frontend/src/components/`
- `frontend/src/hooks/`
- `frontend/src/stores/`
- `frontend/src/types.ts`

観点:

- ストア間で二重管理や不整合がないか
- 非同期処理の競合で古い結果を表示しないか
- 表示追加が既存 UI を壊していないか
- 型定義変更が API 契約に追従しているか

完了条件:

- 主要画面ごとにデータフローを確認
- 追加 UI に対して最低限の動作確認手順を残す

### 5. DevTools / データ / スクリプト

対象:

- `frontend/src/components/devtools/`
- `scripts/fetch_item_sprites.py`
- `data/`
- `templates/items/`
- `docs/implementation/`

観点:

- 開発用機能が本番系コードに不要な影響を出していないか
- 生成物の混入ルールが曖昧でないか
- スクリプト再実行時に再現可能か
- 大量アセット追加に対する管理方針があるか

完了条件:

- 再生成手順または由来を確認
- 不要ファイル混入の有無を確認

## 進め方

1. `docs/review/code-review-progress.md` の対象一覧から、`Not Started` を `In Review` に更新して着手する。
2. レビュー中に見つけた事項は、同ドキュメントの「Findings Log」に追記する。
3. 指摘が修正されたら、該当項目に再確認日と結果を追記する。
4. 全対象が `Done` になったら、未解決事項だけを別途整理してクローズする。

## 判定基準

`Done` としてよい条件:

- 主要リスク観点を一通り確認済み
- 指摘事項が記録済み
- 必要な再確認項目が明記済み
- テスト未実施や確認不能事項があれば明示済み

`Done` にしない条件:

- 読んだだけで結果を残していない
- 指摘を出したが再確認条件が曖昧
- API 契約や状態遷移の確認が途中

## 期待アウトプット

- モジュール別のレビュー完了状況
- 指摘一覧と重要度
- 再確認が必要な項目
- レビュー後に残るリスク
